# 一、整体架构（你现在用的其实是一个“本地代理系统”）

你这个客户端（看起来像 Clash / Mihomo 一类）本质在做三件事：

### 1️⃣ 在你电脑上开一个“本地代理”

比如：

- HTTP 代理：127.0.0.1:7890
- SOCKS 代理：127.0.0.1:7891

👉 所有流量先进入这里（像一个“交通调度中心”）

------

### 2️⃣ 劫持系统流量（关键点）

实现方式有三种（看客户端设置）：

- **TUN 模式（最常见）**
  - 创建虚拟网卡
  - 所有流量先进入这个虚拟网卡
- **系统代理模式**
  - 只代理浏览器/支持代理的软件
- **混合模式**

👉 你现在这类界面，大概率是 **TUN 模式**
 👉 所以：**几乎所有流量都会被它接管**

------

### 3️⃣ 按规则决定“走哪里”

流量进入后，会经历：

```
请求 → 规则匹配 → 选择代理组 → 选节点 → 发出去
```

------

# 二、三个模式的“真实执行逻辑”

## 🟢 规则模式（核心）

不是简单开关，而是：

```
请求 → 匹配规则 → 决定走：
   - 直连
   - 某个代理组
```

### 规则长这样（举例）：

```
DOMAIN-SUFFIX,google.com,Proxy
DOMAIN-SUFFIX,baidu.com,DIRECT
GEOIP,CN,DIRECT
MATCH,Proxy
```

### 实际执行：

| 访问         | 结果         |
| ------------ | ------------ |
| google.com   | 走代理       |
| bilibili.com | 直连         |
| 不认识的网站 | 默认走 Proxy |

👉 所以它是“智能分流系统”

------

## 🔴 全局模式（强制代理）

直接跳过规则：

```
所有请求 → Proxy组
```

👉 等价于：

```
MATCH,Proxy
```

没有判断、没有分流。

------

## ⚪ 直连模式（完全绕过代理）

```
所有请求 → DIRECT
```

👉 相当于：

- 不用节点
- 不走代理协议
- 客户端基本“空跑”

------

# 三、你界面里的“代理组”到底是什么？

你图里这些：

- 自动选择
- 故障转移
- 香港03专线
- 日本01专线

👉 这些不是节点，而是 **策略组（Proxy Group）**

------

## 1️⃣ 节点（最底层）

比如：

- 香港03
- 日本01
- 新加坡01

👉 每个节点 = 一台远程服务器

------

## 2️⃣ 代理组（中间层调度器）

比如：

### 🔵 自动选择（URLTest）

原理：

```
定时访问一个测试网址（如 google.com）
测延迟 → 选最快节点
```

👉 你图里 “香港03 138ms” 就是测出来的

------

### 🟡 故障转移（Fallback）

原理：

```
按顺序试节点：
香港01 → 日本01 → 新加坡01
哪个能连就用哪个
```

👉 不一定最快，但保证“能用”

------

### 🟢 手动选择（Selector）

原理：

```
你手动点哪个，就一直用哪个
```

------

# 四、完整数据流（最重要）

假设你访问：`youtube.com`

在【规则模式】下：

```
1. 请求进入 TUN 虚拟网卡
2. 进入 Clash 内核
3. 匹配规则：
   youtube.com → Proxy
4. Proxy 对应：自动选择
5. 自动选择选出：香港03
6. 使用协议（VMess）连接香港03服务器
7. 服务器帮你访问 youtube
8. 返回数据给你
```

------

# 五、为什么有“延迟 / Timeout / 808”

你图里这些状态：

- `138` → 延迟 ms（正常）
- `Timeout` → 连不上
- `808` → HTTP测试返回码（通常异常）

👉 这些是 **代理组在做测速/探活**

------

# 六、核心原理总结（一句话版本）

👉 整个系统本质是：

```
本地流量 → 被劫持 → 规则判断 → 代理组决策 → 节点执行 → 远程转发
```

------

# 七、你现在这套结构的层级关系

```
[模式]
  ↓
[规则系统]
  ↓
[代理组（自动选择/故障转移）]
  ↓
[节点（香港/日本/...）]
  ↓
[远程服务器]
```

------

# 八、为什么有时候“切全局就好了”

因为：

👉 问题往往出在：

- 规则没匹配到
- DNS 解析错误
- 某些 App 不走规则

👉 全局模式 = 绕过所有判断，直接走代理

------

# 九、进阶一点（你可能会关心）

## DNS 在这里也被接管

客户端通常会：

- 劫持 DNS 请求
- 防止 DNS 泄漏
- 做“国内外分流解析”

------

## 为什么有 UDP / VMess / Reality

这些是 **传输协议层**，决定：

- 抗封锁能力
- 延迟
- 稳定性

------

# 十、给你一个实用建议

你现在这样用就很好：

- 日常：✅ 规则模式 + 自动选择
- 不正常：👉 切全局测试
- 节点不稳：👉 用故障转移
